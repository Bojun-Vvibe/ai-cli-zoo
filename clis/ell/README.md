# ell

> Snapshot date: 2026-04. Upstream: <https://github.com/MadcowD/ell>
> (PyPI distribution: `ell-ai`; binary surface: `ell-studio`).

A Python prompt-engineering framework whose CLI surface is
`ell-studio` — a local web UI backed by a SQLite store that
**versions every prompt and every LLM call** like checkpoints in
a training loop. The "language model program" abstraction
(`@ell.simple` / `@ell.complex` decorators on Python functions)
sits in the same conceptual space as
[`magentic`](../magentic/) and [`pydantic-ai`](../pydantic-ai/),
but `ell`'s differentiator is the **observability + versioning
layer baked into the framework itself**, not bolted on.

## 1. Install footprint

- `pip install -U "ell-ai[all]"` (note: PyPI name is `ell-ai`,
  the import name is `ell`, the CLI is `ell-studio`).
- Pulls SQLAlchemy + FastAPI + the configured provider SDKs
  (~80 MB resolved with `[all]`).
- Local store at `./logdir` (or whatever `--storage` points at);
  `ell-studio --storage ./logdir` boots a web UI on
  `127.0.0.1:5555`.
- No daemon by default — the framework runs in-process inside
  your Python program; `ell-studio` is the optional viewer.

## 3. Models supported

OpenAI (chat + responses), Anthropic, Gemini (AI Studio +
Vertex), Groq, Mistral, Cohere, Bedrock, Ollama, any
OpenAI-compatible endpoint via the OpenAI client's `base_url`.
Provider selection is per-decorator: `@ell.simple(model="gpt-4o")`
or `@ell.simple(model="claude-3-5-sonnet-latest")`. Multimodal
inputs (PIL `Image`, audio) are first-class for providers that
support them.

## 4. MCP support

None as of v0.0.17. Tools in `@ell.complex` are typed Python
callables; there is no MCP-server adapter for mounting external
MCP toolsets, and no first-party MCP-server publishing of an
`ell` program (contrast with [`pydantic-ai`](../pydantic-ai/)'s
`Agent.to_mcp_server()`).

## 5. Sub-agent model

None as a first-class concept. `@ell.complex` supports tool
loops (model emits a tool call → Python executes → result feeds
back), but multi-agent composition is just "one Python function
calls another" — no managed crew, no role registry, no shared
message bus. If you want a manager-LLM-spawning-worker-LLMs
shape, see [`crewai`](../crewai/) or [`metagpt`](../metagpt/).

## 6. Telemetry stance

Off, no analytics. Egress is exclusively to the configured LLM
provider. The local SQLite store at `./logdir` is the *opposite*
of telemetry: every prompt + completion + token count + latency +
serialised function source is written locally and viewable in
`ell-studio`. Sensitive prompts stay on disk; nothing leaves
unless you point `ell-studio` at a remote storage backend (which
is opt-in and not the default).

## 7. Prompt-cache strategy

Provider-native (Anthropic prompt caching, OpenAI prefix caching)
flows through unchanged because `ell` does not rewrite the
provider request payload. No `ell`-side response cache —
re-invoking the same `@ell.simple` function makes a fresh API
call and creates a new versioned row in the local store (this is
intentional: it is a checkpointing system, not a memo).

## 8. Hot keybinds / commands

- `ell-studio --storage ./logdir` — boot the local web UI on
  `127.0.0.1:5555`; lists every "language model program" that has
  been called, every version of each, and a side-by-side diff
  viewer.
- `ell-studio --storage ./logdir --port 8000` — pick a port.
- The Python surface is the actual API: `@ell.simple` for plain
  string-out functions, `@ell.complex` for tool-using or
  multimodal functions, `ell.init(store='./logdir', autocommit=True)`
  to enable on-disk versioning + autocommit messages from a
  small model.

## 9. Killer feature, weakness, when to choose

**Killer feature.** **Prompts are versioned like code, with a
local web UI to read the history.** `ell.init(autocommit=True)`
turns every `@ell.simple` function call into a checkpoint — the
function's source, the resolved prompt, the model's response,
and an autogenerated commit message all land in SQLite. Open
`ell-studio` to see the diff between version N-1 and N of any
LMP, exactly when each version ran, and what the model said.
This is the answer to "I improved my prompt last Tuesday and now
I cannot remember what I changed" — every other catalog peer
asks you to wire up Langfuse / Phoenix / OTel by hand.

**Weakness.** **Pre-1.0, narrow agent surface, no MCP.** The
package is `0.0.x`, the API still moves, and the framework is
**not** an agent runtime — there is no built-in tool loop with
sandboxing, no MCP client/server, no multi-agent orchestration.
If you need a coding agent, `ell` is the wrong layer; pair it
with one of the agent CLIs in this catalog instead. Also: the
`ell-studio` UI requires running a local web server, which is a
heavier observability story than "tail a JSONL file".

**When to choose.**

- You are **iterating on prompts** for a Python program and want
  **automatic version control + a viewer** without bolting on
  Langfuse / Phoenix / Helicone.
- You want the **type-shape ergonomics of a Python decorator**
  (similar to `magentic`'s `@prompt`) **plus** a baked-in trace
  store.
- You are doing **research-style prompt engineering** where
  reproducing "what did the prompt look like on Tuesday" matters
  as much as the model output itself.
- You want **first-class multimodal inputs** (PIL images, audio)
  inline in the function signature.

**When not to choose.**

- You need **typed Pydantic-validated outputs** as the load-bearing
  contract (→ [`pydantic-ai`](../pydantic-ai/),
  [`magentic`](../magentic/)).
- You need an **agent loop** with a tool sandbox or MCP — `ell`
  is a prompt framework, not an agent runtime
  (→ [`smolagents`](../smolagents/), [`crewai`](../crewai/),
  [`pydantic-ai`](../pydantic-ai/)).
- You need a **stable 1.x API** for a production service — the
  package is still 0.0.x as of 2026-04, breaking changes are
  expected.
- You want **observability without a local web server** — `ell`'s
  story is `ell-studio`, which needs a port.

## Comparable tools in this catalog

- [`magentic`](../magentic/) — Python peer with the `@prompt`
  decorator shape; structured-output via Pydantic, no built-in
  trace store.
- [`pydantic-ai`](../pydantic-ai/) — heavier Python framework
  with type-safety + MCP server/client + Logfire integration; an
  agent runtime, not just a prompt framework.
- [`promptfoo`](../promptfoo/) — eval harness; complementary, not
  competing — `promptfoo` regression-tests prompts, `ell` versions
  them as you iterate.
- [`llm`](../llm/) — Simon Willison's Python CLI; logs every
  prompt/response to a SQLite store too, but at the *CLI*
  invocation level, not the in-process function level.
